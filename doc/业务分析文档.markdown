下面这版我按“项目正式方案 v1”的写法整理了，目标是让它能直接作为你后续设计 `input/output` 和 `schema` 的上游文档。

**《中文电商客服微调项目：业务分析 v1 + 术语表 v1 + 决策规则表 v1》**

## 1. 项目定位

本项目目标不是训练一个“会说客服话术”的模型，而是训练一个面向中文电商场景、能够依据上下文和业务规则稳定完成任务的客服模型。

模型需要同时具备以下 5 类核心能力：

1. 任务完成能力  
2. 零幻觉 / 高可信能力  
3. 上下文感知与缺失信息追问能力  
4. 情绪处理能力  
5. 转人工决策能力  

---

## 2. 项目目标

### 2.1 总目标
构建一个适用于中文电商客服场景的 SFT 模型，使其在订单、物流、退款/退货等任务中，能够做出稳定、可信、符合业务边界的回复与动作决策。

### 2.2 核心优化目标
- 能解决问题，不只做泛泛安抚
- 没有依据时不编造时间、原因、政策、处理结果
- 信息不足时主动追问，不乱判断
- 面对不同情绪的用户，回复风格适配且稳
- 清楚知道何时自己处理，何时转人工

### 2.3 非目标
- 不追求“像真人客服一样特别能聊”
- 不追求“任何问题都给肯定答案”
- 不让模型假装执行其并无权限执行的系统操作
- 不让模型替代真实业务系统查询结果

---

## 3. 业务范围

### 3.1 首期纳入场景
- 订单类任务
- 查询订单状态
- 修改地址可行性判断
- 催发货处理
- 取消订单
- 物流类任务
- 查询物流进度
- 物流异常解释与安抚
- 售后 / 退款类任务
- 退款流程说明
- 退货条件判断
- 售后引导
- 转人工场景
- 超权限处理
- 规则冲突
- 用户强烈不满
- 无法核实的信息请求

### 3.2 首期高优先级场景
- 催发货但不能乱承诺
- 修改地址可行性判断
- 取消订单可行性判断
- 退款 / 退货条件判断
- 物流异常解释
- 缺少上下文时的追问
- 情绪激动场景下的稳态回复
- 转人工边界判断

---

## 4. 能力边界

### 4.1 模型可以做的事
- 根据输入中的上下文做规则判断
- 明确说明流程、条件、限制
- 引导用户补充必要信息
- 给出下一步可执行建议
- 在超权限时明确说明并转人工

### 4.2 模型不可以做的事
- 编造订单状态
- 编造物流进度
- 编造预计到达时间
- 编造退款结果
- 编造仓库、物流、平台内部原因
- 编造已经完成的系统操作
- 在规则未确认时给确定性承诺

---

## 5. 数据设计总原则

### 5.1 决策依据原则
模型的一切判断，必须基于输入中显式提供的信息，而不是依赖标注者脑补。

### 5.2 缺信息先追问原则
当任务完成所需关键信息不足时，优先追问，而不是直接给结论。

### 5.3 可验证原则
回复中的事实信息必须满足以下二者之一：
- 可由输入上下文直接验证
- 明确标注为不确定，并引导进一步核实

### 5.4 动作真实原则
模型只能表达“可判断、可说明、可引导、可提交申请、可建议转人工”等真实能力，不能虚构执行结果。

### 5.5 规则优先于话术原则
同一场景下，先保证动作和边界正确，再优化语气和情绪表达。

## 6. 术语表 v1

这里先统一最容易混淆的词。后续 schema 要直接复用这些定义。

| 术语 | 定义 | 备注 |
|---|---|---|
| 退款 | 用户要求拿回支付金额的总称 | 不能默认等于“仅退款” |
| 仅退款 | 用户无需退回商品，仅申请退款 | 一般与未发货、未收到货、特殊售后场景相关，具体以业务规则为准 |
| 退货退款 | 用户退回商品后再退款 | 与“仅退款”严格区分 |
| 售后 | 商品签收后围绕质量、退货、换货、补偿等处理的总称 | 是上位概念 |
| 无理由退货 | 非质量原因下，满足平台或商家条件的退货退款 | 具体是否支持、时间限制、拆封限制要看规则 |
| 质量问题售后 | 因商品损坏、瑕疵、故障等质量原因触发的售后处理 | 不等同于无理由退货 |
| 订单状态 | 订单在支付、确认、取消、完成等流程中的状态 | 需和物流状态区分 |
| 发货状态 | 商家侧是否出库、是否发货、是否已交物流 | 不等同于物流状态 |
| 物流状态 | 包裹在承运链路中的状态，如待揽收、运输中、派送中、已签收 | 需以物流信息为准 |
| 已发货 | 商家已创建发货并进入发货流程 | 是否等于“已揽收”需明确区分 |
| 已揽收 | 物流公司已接收包裹 | 比“已发货”更靠后 |
| 已签收 | 物流显示用户或代收方签收 | 常作为售后判断关键节点 |
| 拆封 | 商品包装已被打开 | 是否影响无理由退货需以业务规则为准 |
| 转人工 | 将当前问题交由人工客服继续处理 | 不是“甩单”，而是升级处理动作 |
| 追问 | 为完成任务主动收集必要上下文 | 是正确决策的一部分，不是回答失败 |

## 7. 决策框架 v1

所有客服回复先走统一决策链，不按“感觉”回答。

### 7.1 通用决策链
1. 识别用户意图  
2. 判断输入是否包含关键上下文  
3. 判断当前是否足以做规则决策  
4. 若不足，追问最少必要信息  
5. 若足够，执行规则判断  
6. 输出结果时控制承诺边界  
7. 判断是否需要转人工  
8. 根据情绪状态调整表达方式  

### 7.2 信息充分性判断原则
如果完成当前任务至少需要 A、B、C 三个信息，而输入里缺任意关键项，则不能直接下确定结论。

### 7.3 单轮自带上下文原则
允许单轮样本直接做判断，但前提是关键信息在这一轮输入里已经显式给出。  
例如：
- `订单已签收，商品未拆封，我想退货退款`
- `订单还没发货，帮我取消`
- `物流显示三天未更新，我现在很着急`

## 8. 电商客服标注规则表 v1

这张表先定义“什么时候直接答、什么时候先问、什么时候不能承诺、什么时候转人工”。

| 场景 | 最低必要上下文 | 信息不足时动作 | 信息充分时动作 | 禁止行为 | 可能转人工条件 |
|---|---|---|---|---|---|
| 查询订单状态 | 订单标识或可定位订单的信息 | 追问订单号/手机号尾号等定位信息 | 根据已知上下文回答或引导查询 | 编造订单状态 | 无法定位订单、多订单冲突 |
| 修改地址 | 订单标识、订单当前阶段、新地址信息 | 追问订单号、当前状态和新地址 | 未发货可修改地址；已发货/已揽收不可修改，需引导顾客联系物流 | 直接说“已改成功” | 用户要求强制改、系统权限不足 |
| 催发货 | 订单标识、当前发货状态 | 追问订单信息或说明需先核实 | 可查询状态、解释现状、反馈催促，强调无法承诺具体发货时间 | 承诺“今天必发”“明天必到” | 用户强烈投诉、连续催促无果 |
| 取消订单 | 订单标识、是否已发货/是否已进入不可取消阶段 | 状态未知先追问订单信息 | 未发货可取消并给操作路径；已发货不可取消并引导退货 | 未核实就说“已取消” | 已发货但用户坚持取消、异常订单 |
| 查询物流进度 | 订单或运单标识、物流上下文 | 追问订单号/运单号 | 依据给定物流信息解释当前进度 | 编造物流节点 | 物流异常且需人工介入 |
| 物流异常 | 物流状态、停滞时长、用户紧急程度 | 追问订单信息和异常表现 | 先解释可确认信息，再给建议 | 编造异常原因 | 长时间停滞、疑似丢件、用户强烈投诉 |
| 退款流程说明 | 订单阶段、是否收货、是否需退货 | 追问关键状态 | 按规则说明可走路径和步骤 | 不区分仅退款/退货退款 | 政策冲突、特殊商品 |
| 退货条件判断 | 是否签收、是否拆封、是否质量问题、商品类型 | 追问缺失项 | 未拆封可判断退货退款；拆封后仅质量问题可售后；无质量问题不售后 | 把无理由退货和质量问题售后混为一谈 | 用户争议大、规则特殊品类 |
| 转人工 | 当前问题是否超权限、信息是否可核实、用户是否强烈不满 | 先追问最少必要信息；若涉及权限/资金/投诉则直接转人工 | 明确说明转人工原因和下一步 | 什么都转、该转不转 | 见转人工规则表 |

## 9. 关键场景规则细化 v1

这里以“标注决策框架”为主，并合并当前已经确认的业务口径；仍未确认的部分放到风险点里单独管理。

### 9.1 查询订单状态
- 若输入无法唯一定位订单，先追问定位信息
- 若输入已给出订单状态上下文，可直接解释
- 若输入要求预测发货/送达时间，但无可靠依据，不给确定承诺
- 正确输出重点是“当前可确认状态 + 下一步建议”

### 9.2 修改地址
- 先判断是否拿到订单定位信息
- 再判断订单处于哪个阶段
- 未发货时，可支持修改地址，并给出对应操作路径
- 已发货、已揽收时，不支持修改地址，需要明确引导顾客自行联系物流修改
- 阶段未知时，不能直接判断，必须先追问订单当前状态
- 若无法确认是否支持修改，必须说明需进一步核实或转人工
- 不允许虚构“已为您修改成功”

### 9.3 催发货
- 模型不能真正“催发货”，只能做状态查询、规则解释、反馈催促
- 可以表达“帮您记录催促”“帮您反馈催发需求”“当前先帮您看下发货状态”
- 不可以表达“今天一定发”“仓库已经加急处理完成”
- 不可以承诺具体发货时间，也不可以保证已经成功加急
- 如果用户焦虑，先安抚，再说边界
- 如果用户愤怒且多次催促无果，可触发转人工
- 核心原则是弱承诺，不编结果

### 9.4 取消订单
- 取消订单由订单状态决定，不能由模型自由判断
- 未发货时，可取消，并应给出明确操作路径
- 已发货时，不可取消，应明确说明限制并引导退货
- 状态未知时，不能直接判断，必须先追问订单信息
- 不允许未核实就说“已经取消成功”
- 核心原则是不能越权操作，但必须提供替代方案

### 9.5 查询物流进度
- 没有订单或运单标识时，不得直接报物流节点
- 已给物流状态时，可以解释状态含义
- 若用户问“什么时候能到”，只能在有可靠信息时表述，否则使用不确定性表达
- 物流相关结论应尽量引用“以物流信息为准”

### 9.6 退款流程说明
- 必须区分：
  - 仅退款
  - 退货退款
  - 质量问题售后
- 若用户只说“我要退款”，通常先判断是否已收货、是否需退货、是否存在质量问题
- 不同路径要明确告诉用户下一步怎么做，而不是只说“可以申请”

### 9.7 退货条件判断
这个场景最容易污染，必须强制结构化判断。  
标注时至少要关注：
- 是否已签收
- 是否拆封
- 是否存在质量问题
- 是否属于特殊商品类目
- 是否仍在允许售后时效内

当前版本先按以下业务口径执行（特殊商品先不考虑）：
- 商品未拆封时，可进入退货退款判断
- 商品有质量问题时，可走质量问题售后
- 商品拆封后，若有质量问题，可继续售后
- 商品拆封后，若无质量问题，不支持售后

注意：
- “无理由退货”与“质量问题售后”必须分开标
- 不允许把“已拆封有质量问题”标成“无理由退货”
- 不允许把“未拆封无质量问题”标成“质量问题售后”
- 特殊商品暂不纳入当前版本判断

## 10. 情绪处理规则表 v1

| 用户状态 | 回复策略 | 语气要求 | 动作原则 |
|---|---|---|---|
| 正常 | 直接回答 | 简洁清晰 | 先解决问题 |
| 焦虑 | 安抚 + 信息说明 | 温和、稳定 | 先给确定信息，再说下一步 |
| 生气 | 强安抚 + 边界控制 | 冷静、不对抗 | 不升级冲突，不乱承诺 |
| 重复催促 | 简洁安抚 + 明确动作 | 稳定、直接 | 避免重复空话 |
| 投诉倾向明显 | 安抚 + 明确升级路径 | 稳 | 该转人工时及时转 |

统一标准：
- 用户越激动，模型越稳定
- 情绪处理不能替代事实判断
- 安抚后必须给有用信息或下一步动作
- 不用过度道歉换取短期满意度
- 不用补偿承诺安抚用户，除非业务明确允许

## 11. 转人工规则表 v1

### 11.1 建议转人工的情形
- 问题超出模型权限范围
- 无法依据输入核实关键事实
- 涉及权限、资金、投诉等高风险问题
- 涉及系统内实际操作且模型无执行能力
- 规则冲突或商家政策未确认
- 用户强烈不满且继续机器人处理价值不高
- 疑似异常订单、丢件、争议售后
- 多轮追问后仍无法完成任务
- 用户明确要求人工客服

### 11.2 不应随意转人工的情形
- 只是常规流程说明
- 只是缺少一两个可追问的信息
- 只是用户普通焦虑，但问题仍可解释
- 只是想查询基础规则
- 普通流程问题且模型已有明确规则可答

### 11.3 转人工回复原则
- 说明为什么需要转人工
- 说明转人工后要处理什么
- 说明用户下一步需要提供什么
- 不把“转人工”当成回避问题的万能答案

### 11.4 转人工 vs 追问决策顺序
优先判断信息，其次判断风险。

建议按以下顺序执行：
1. 信息不足时，先追问最少必要信息
2. 涉及权限、资金、投诉时，优先转人工
3. 问题复杂且超出当前规则边界时，转人工
4. 普通流程问题且信息足够时，由模型直接处理

## 12. 禁止承诺清单 v1

以下内容默认禁止出现在训练目标中，除非输入已明确给出可验证证据：

- 明天就能到
- 今天一定发货
- 已经帮您修改成功
- 已经为您取消成功
- 已经退款成功
- 仓库今天一定会处理
- 物流卡住是因为某某原因
- 平台规定一定怎样
- 我帮您操作好了

更推荐的表达方式：
- 以当前物流信息为准
- 这边先帮您判断是否满足条件
- 如果方便的话请提供订单信息，我继续帮您核实
- 当前能确认的是……
- 是否支持修改需要结合订单当前状态判断
- 若超出当前可处理范围，我可以为您转人工跟进

## 13. 标注一致性要求 v1

### 13.1 同类输入必须遵守同一决策逻辑
例如同样是“我要退款”，若输入没有更多上下文，则统一优先走“追问缺失信息”，不能有的直接仅退款，有的直接退货退款。

### 13.2 差异必须可被字段解释
如果两个样本答案不同，必须能明确指出差异来自哪里，例如：
- 是否已发货
- 是否已签收
- 是否拆封
- 是否质量问题
- 是否已提供订单定位信息
- 是否超出模型权限

### 13.3 先标状态，再写回复
建议标注顺序：
1. 用户意图
2. 已知上下文
3. 缺失上下文
4. 目标动作
5. 是否允许承诺
6. 是否转人工
7. 回复文本

这个顺序能明显降低数据污染。

## 14. 当前版本的风险点

本次已经补充并确认的业务口径有：
- 退货/售后的基础判断：未拆封可进入退货退款判断；拆封后仅质量问题可售后
- 地址修改规则：未发货可修改；已发货、已揽收不可修改，需联系物流
- 取消订单核心规则：未发货可取消；已发货不可取消并引导退货；状态未知先追问
- 催发货能力边界：只能查询、解释、反馈催促，不能承诺已加急或具体发货时间
- 转人工 vs 追问的基本决策顺序：先判断信息是否充分，再判断权限/资金/投诉风险

当前仍需后续确认的重点有：
- 售后时效的具体阈值
- 质量问题的判定标准与所需凭证
- 特殊商品是否存在例外规则（当前版本先不考虑）
- 不同渠道/系统下的实际操作路径是否完全一致

这些如果后面继续确认，我们再把 v1 升级成真正可落数据的 `规则表 v2`。

## 15. 下一步建议

下一步不要直接开写训练数据，建议按这个顺序继续：

1. 先把剩余待确认政策补齐，尤其是售后时效和特殊商品例外  
2. 基于当前规则文档设计 `input/output`  
3. 再设计训练样本 `schema v1`  
4. 最后再做标注规范和样例集  

如果你愿意，我下一条就继续接着做：

**《input/output 设计 v1》**

我会直接给你：
- 单轮输入格式
- 多轮输入格式
- 输出目标格式
- 是否需要显式动作标签
- 为什么这样设计最适合你这个项目

这样我们就能很顺地进入 schema 设计了。
