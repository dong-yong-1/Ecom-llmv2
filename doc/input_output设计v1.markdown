# 《中文电商客服微调项目：input/output 设计 v1》

## 1. 设计目标

本项目当前阶段的目标，不是让模型输出复杂结构化指令，而是让模型在中文电商客服场景中，基于给定上下文，稳定产出一条：

- 礼貌
- 专业
- 不编造
- 能处理常规流程
- 信息不足时会追问
- 必要时会转人工

的客服回复。

因此，本项目的 `input/output` 设计采用：

- `Input = 系统指令 + 上下文（可选） + 对话历史`
- `Output = 当前轮客服回复`

这是一种偏 chat-style SFT 的设计，优点是贴近真实业务使用方式，同时便于训练和推理落地。

---

## 2. 为什么采用这种设计

### 2.1 优点
- 形式简单，易于构造训练样本
- 和真实客服对话形式接近
- 输出目标明确，模型只需要学习“当前这轮怎么回”
- 适合先做第一版 SFT，快速验证效果

### 2.2 风险
- 如果只保留自然语言回复，模型容易学到话术，但不一定稳定学会决策
- “什么时候追问”“什么时候转人工”“什么时候不能承诺”这类能力，单靠 system prompt 不够
- 出现 bad case 时，仅看回复文本不容易定位是规则错、上下文错，还是标注错

### 2.3 设计结论
因此推荐采用“两层设计”：

- 训练接口层：`Input = 系统指令 + 上下文 + 对话历史`，`Output = 当前轮客服回复`
- 数据管理层：额外保存辅助标注字段，供标注、质检、评测使用，但不强制要求模型输出

也就是说，模型看到和学习的仍然是自然语言客服回复，但数据底层需要保留更强的结构化信息。

---

## 3. 模型目标如何映射到 input/output

当前项目已经明确的核心目标可以映射为：

| 模型目标 | 在输入输出设计中的体现 |
|---|---|
| 任务完成能力 | 上下文中提供任务判断所需信息，输出要求给出可执行下一步 |
| 零幻觉 / 高可信 | system 明确禁止编造；上下文中只放可确认事实 |
| 上下文感知与追问 | 缺失关键信息时，让模型通过回复主动追问 |
| 情绪处理能力 | 在对话历史中体现用户情绪，输出中控制语气 |
| 转人工决策能力 | 对高风险、超权限、投诉类问题，在回复中体现转人工引导 |

换句话说，这个项目的关键不是让模型“会聊天”，而是让模型基于输入信息作出正确客服决策，并把该决策表达成自然语言回复。

---

## 4. Input 设计 v1

推荐统一输入结构为三段：

1. 系统指令
2. 上下文
3. 对话历史

### 4.1 系统指令

系统指令建议固定为一段高约束、低歧义的说明，不要每条样本都写得不一样。

推荐版本：

```text
你是电商客服助手，需要礼貌、专业、基于已知信息回答，不编造订单状态、物流进度、处理结果或承诺时间；信息不足时先追问，超出权限或高风险场景时转人工。
```

这一版 system 的作用是统一模型边界，而不是替代具体业务规则。

### 4.2 上下文

上下文是模型本轮可使用的业务事实。  
建议使用“结构化字段 + 显式未知值”的方式，而不是完全自然语言描述。

推荐原则：
- 已知信息就明确填写
- 未知信息就写 `未知`
- 不要靠省略字段表达未知
- 尽量使用稳定字段名，减少标注漂移

### 4.3 对话历史

对话历史用于承载：
- 用户当前意图
- 已经追问过的信息
- 用户情绪
- 多轮中的上下文变化

推荐保留最近几轮必要历史，不建议无节制拼接长对话。

---

## 5. 上下文字段设计建议 v1

字段不要一次塞太多，否则会增加标注成本和噪声。  
建议分成两类：

### 5.1 通用核心字段

这些字段适合大多数订单、物流、售后场景：

| 字段名 | 含义 | 典型取值 |
|---|---|---|
| `order_status` | 订单状态 | `未知` / `待支付` / `已支付` / `已取消` / `已完成` |
| `shipping_status` | 发货状态 | `未知` / `未发货` / `已发货` |
| `logistics_status` | 物流状态 | `未知` / `待揽收` / `已揽收` / `运输中` / `派送中` / `已签收` |
| `is_signed` | 是否签收 | `未知` / `是` / `否` |
| `is_opened` | 是否拆封 | `未知` / `是` / `否` |
| `has_quality_issue` | 是否有质量问题 | `未知` / `是` / `否` |
| `order_id_provided` | 是否已提供可定位订单信息 | `未知` / `是` / `否` |

说明：
- `shipping_status` 和 `logistics_status` 建议分开，不要混成一个字段
- `is_signed` 虽然和 `logistics_status=已签收` 有关联，但单独保留更利于售后场景判断
- 你已经明确说当前版本不需要 `是否特殊商品`，那这一版先不放

### 5.2 场景扩展字段

只在需要的场景里增加，不必每条样本都出现：

| 字段名 | 适用场景 | 典型取值 |
|---|---|---|
| `user_request` | 所有场景 | `取消订单` / `催发货` / `修改地址` / `退款` / `退货退款` |
| `new_address_provided` | 修改地址 | `未知` / `是` / `否` |
| `logistics_delay_days` | 物流异常 | 数字或 `未知` |
| `user_emotion` | 情绪处理 | `正常` / `焦虑` / `生气` / `投诉倾向` |
| `need_after_sales` | 售后相关 | `未知` / `是` / `否` |

原则是：
- 核心字段稳定
- 扩展字段按场景补充
- 不为追求完整而加入无用字段

## 6. Output 设计 v1

`Output` 只保留当前轮客服回复，这个方向是成立的，建议继续采用。

输出要求：
- 必须是自然语言客服回复
- 必须基于输入中已知事实
- 不允许编造订单、物流、退款、加急、修改成功等结果
- 信息不足时，要在回复里体现追问
- 超权限或高风险问题，要在回复里体现转人工

### 6.1 Output 的优点
- 训练目标简单
- 样本格式统一
- 更贴近真实部署时的最终输出

### 6.2 Output 的补充建议

虽然模型输出只保留客服回复，但数据集内部仍建议保留辅助字段，例如：

- `intent`
- `decision_type`
- `missing_slots`
- `should_transfer`
- `policy_basis`

这些字段不要求模型生成，但强烈建议在数据层保留。

---

## 7. 推荐的数据管理结构

推荐把“训练字段”和“辅助标注字段”分开。

### 7.1 训练字段

```json
{
  "system": "...",
  "context": {},
  "history": [],
  "target": "..."
}
```

### 7.2 辅助标注字段

```json
{
  "meta": {
    "intent": "...",
    "decision_type": "...",
    "should_transfer": false,
    "missing_slots": [],
    "policy_basis": []
  }
}
```

这样做的好处是：
- 训练时保持简洁
- 标注时保持一致
- 评测时可以定位问题
- 后续如果要升级为结构化输出，也能平滑迁移

---

## 8. 推荐的 decision_type 枚举

虽然这不是模型输出字段，但建议在标注侧统一枚举，避免样本风格混乱。

推荐值：
- `direct_answer`：信息足够，直接回答
- `ask_followup`：信息不足，先追问
- `rule_explanation`：主要是在解释规则
- `transfer_human`：需要转人工

这个字段的价值在于：
- 防止同类问题有的直接答、有的先问
- 有助于后续做分类评测
- 能帮助筛查“明明该追问却直接答了”的坏样本

## 9. 单轮样本格式建议

### 9.1 文本拼接格式

```text
系统：
你是电商客服助手，需要礼貌、专业、基于已知信息回答，不编造订单状态、物流进度、处理结果或承诺时间；信息不足时先追问，超出权限或高风险场景时转人工。

上下文：
订单状态: 已支付
发货状态: 已发货
物流状态: 运输中
是否签收: 否
是否拆封: 未知
是否质量问题: 未知
是否已提供订单信息: 是

对话：
用户：怎么还没到？

输出：
客服：您好，这边看到您的订单已经发货，当前物流状态是运输中。运输途中具体送达时间还请以最新物流信息为准。如果您愿意，也可以把订单号发我，我继续帮您核实物流进度。
```

### 9.2 JSON 结构格式

```json
{
  "system": "你是电商客服助手，需要礼貌、专业、基于已知信息回答，不编造订单状态、物流进度、处理结果或承诺时间；信息不足时先追问，超出权限或高风险场景时转人工。",
  "context": {
    "order_status": "已支付",
    "shipping_status": "已发货",
    "logistics_status": "运输中",
    "is_signed": "否",
    "is_opened": "未知",
    "has_quality_issue": "未知",
    "order_id_provided": "是"
  },
  "history": [
    {
      "role": "user",
      "content": "怎么还没到？"
    }
  ],
  "target": "您好，这边看到您的订单已经发货，当前物流状态是运输中。运输途中具体送达时间还请以最新物流信息为准。如果您愿意，也可以把订单号发我，我继续帮您核实物流进度。"
}
```

建议底层存 JSON，导出训练时再转成文本模板。

## 10. 多轮样本格式建议

多轮样本的重点不是“轮数越多越好”，而是让模型看到：
- 已经问过什么
- 用户补充了什么
- 现在是否足以做判断

### 10.1 示例：取消订单

```json
{
  "system": "你是电商客服助手，需要礼貌、专业、基于已知信息回答，不编造订单状态、物流进度、处理结果或承诺时间；信息不足时先追问，超出权限或高风险场景时转人工。",
  "context": {
    "order_status": "已支付",
    "shipping_status": "未知",
    "logistics_status": "未知",
    "order_id_provided": "是"
  },
  "history": [
    {
      "role": "user",
      "content": "帮我取消订单"
    },
    {
      "role": "assistant",
      "content": "可以的，我先帮您判断是否还支持取消。请问当前订单是否已经发货？"
    },
    {
      "role": "user",
      "content": "还没发货"
    }
  ],
  "target": "如果订单还未发货，一般是可以取消的。您可以进入订单详情页查看是否有取消入口并按页面提示操作；如果页面没有入口，再联系人工客服进一步处理。"
}
```

### 10.2 多轮设计原则
- 只保留完成当前判断必要的历史
- 助手历史回复也应纳入，因为它影响当前轮判断
- 不建议把大量无关寒暄放进样本
- 多轮样本应重点覆盖“先追问，后回答”的链路

---

## 11. 关于“未知字段”的原则

你已经明确指出这一版不需要 `是否特殊商品: 未知`，这是合理的。

建议原则是：
- 只保留当前业务明确会影响判断的字段
- 不加入当前阶段不打算处理的维度
- 否则会增加标注成本，还会让模型学到无用噪声

因此当前 v1 不放：
- `is_special_product`

等到后续真正处理特殊商品规则时，再升级 schema。

---

## 12. 标注侧必须补的辅助字段

虽然训练输出只有客服回复，但标注侧建议至少补以下字段：

| 字段 | 作用 |
|---|---|
| `intent` | 标注用户当前意图 |
| `decision_type` | 标注这轮是直接答、追问、解释规则还是转人工 |
| `missing_slots` | 标注缺失了哪些必要信息 |
| `should_transfer` | 标注是否应转人工 |
| `policy_basis` | 标注本轮依据的业务规则 |

这是为了避免同类数据出现下列污染：
- 同样信息不足，有的样本追问，有的样本直接答
- 同样是已发货取消，有的说能取消，有的说不能取消
- 同样是拆封无质量问题，有的说能退，有的说不能退

---

## 13. 最推荐的落地方案

基于当前项目阶段，最推荐采用的方案是：

### 13.1 模型训练格式
- Input：系统指令 + 结构化上下文 + 对话历史
- Output：当前轮客服回复

### 13.2 数据底层格式
- 除训练字段外，额外保留 `intent / decision_type / missing_slots / should_transfer / policy_basis`

### 13.3 设计理由
- 训练实现简单
- 模型最终输出自然
- 标注一致性更高
- 评测和错误分析更容易
- 未来升级空间更大
